## 📖 主体讲稿

### Slide 1 — 先建立今天的直觉

把今天的内容想成一个工具箱。遇到题目时，先问三件事：它在解决什么问题？输入是什么？输出应该长什么样？只要这三件事想清楚，选择题、连线题和开放实战题都会变简单。

### Slide 2 — 考点 1-4：逐题白话拆解

- 当题目问「类比：API Key 相当于外卖场景中的什么？」时，正确理解是：会员卡/身份验证——证明你有权限使用这个服务。
- 连线题要记成配对表：Endpoint（端点） = API 的网络地址；API Key = 身份验证密钥；Request（请求） = 发送给 AI 的数据；Response（响应） = AI 返回的结果。
- 当题目问「安全：API Key 不小心上传到 GitHub 会怎样？」时，正确理解是：会被恶意脚本扫描到并盗用——可能产生巨额费用。
- 当题目问「安全：API Key 应该存储在 ___ 文件中，而不是直接写在代码里。」时，正确理解是：.env（环境变量文件）。

讲的时候提醒学员：不要只背关键词，要把关键词放回真实场景里。题目一换皮，场景理解才是拿分关键。

### Slide 3 — 考点 5-8：逐题白话拆解

- 当题目问「成本：为什么输出 Token 比输入 Token 贵？」时，正确理解是：生成每个输出 Token 需要更多计算资源（逐个预测下一个词）。
- 当题目问「省钱：怎么控制 API 成本？」时，正确理解是：选择合适的模型（简单任务用便宜的小模型）+ 设置月度预算上限 + 缓存重复请求。
- 开放实战要讲清楚：直接写 Prompt：围绕「公开 Web 产品集成 OpenAI API Key」，写一个可以直接复制给大模型使用的 API 调用与安全 Prompt。要求包含：角色/目标、输入信息、执行步骤、输出格式、边界限制。 评分时看这些点：是否能直接复制使用；是否围绕 API 调用与安全 和「公开 Web 产品集成 OpenAI API Key」展开；是否包含角色/目标、输入、步骤、输出格式和边界；是否能降低「前端泄露 Key 或账单暴增」风险；语言是否清晰不绕。
- 开放实战要讲清楚：结构化表达：为「公开 Web 产品集成 OpenAI API Key」写一份可直接复制使用的任务说明。必须面向「Web 开发者」，并明确输入、处理步骤、输出格式和禁止事项。 评分时看这些点：结构是否清晰；输入/步骤/输出是否完整；是否符合 Web 开发者 的理解水平；禁止事项是否能降低「前端泄露 Key 或账单暴增」风险。

讲的时候提醒学员：不要只背关键词，要把关键词放回真实场景里。题目一换皮，场景理解才是拿分关键。

### Slide 4 — 考点 9-12：逐题白话拆解

- 开放实战要讲清楚：边界处理：列出「公开 Web 产品集成 OpenAI API Key」在真实使用中最容易失败的 4 种情况，并分别写出 AI 应该如何回应或兜底。 评分时看这些点：是否包含真实边界场景；每个兜底是否具体可执行；是否避免让 AI 编造；是否覆盖「前端泄露 Key 或账单暴增」这个核心风险。
- 开放实战要讲清楚：输出格式设计：为「API 安全清单」设计一个标准输出模板。要求模板字段清楚、顺序合理，并说明每个字段为什么需要。 评分时看这些点：模板是否能支撑 至少 5 条安全措施和后果；字段是否完整且不冗余；是否说明字段用途；是否便于后续复制、评审或程序解析。
- 开放实战要讲清楚：质量评审：假设另一个同学提交了「API 安全清单」。请写一份评分标准，至少包含 5 个维度，并给出每个维度的满分表现和扣分点。 评分时看这些点：是否有 5 个以上评分维度；维度是否贴合 API 调用与安全；是否区分满分表现和扣分点；是否能指导同学改进。
- 开放实战要讲清楚：示例驱动：为「公开 Web 产品集成 OpenAI API Key」补充 2 组高质量示例：一组是简单输入，一组是困难输入。每组都要写出理想输出，并解释困难输入难在哪里。 评分时看这些点：是否有简单和困难两组示例；输入输出是否配套；困难点解释是否准确；示例是否能帮助大模型学会 API 调用与安全。

讲的时候提醒学员：不要只背关键词，要把关键词放回真实场景里。题目一换皮，场景理解才是拿分关键。

### Slide 5 — 考点 13-15：逐题白话拆解

- 开放实战要讲清楚：约束优化：把「API 安全清单」压缩成一个更稳的版本：要求减少废话，但不能丢掉角色、任务、边界、输出格式和检查标准。 评分时看这些点：是否保留关键约束；是否去掉空话；是否更容易执行；是否仍能防止「前端泄露 Key 或账单暴增」。
- 开放实战要讲清楚：真实迁移：把今天的「API 调用与安全」迁移到你自己的一个工作/学习场景。请写出场景、目标用户、任务输入、期望输出和一版初始方案。 评分时看这些点：场景是否真实具体；是否能体现 API 调用与安全；输入输出是否明确；方案是否可执行；是否说明如何评估好坏。
- 开放实战要讲清楚：综合大题：完整产出一份「API 安全清单」，用于「公开 Web 产品集成 OpenAI API Key」。必须包含：1）背景和目标；2）面向对象：Web 开发者；3）完整方案正文；4）输出格式或交付物；5）至少 3 个质量检查点；6）针对「前端泄露 Key 或账单暴增」的风险预案。要求写到可以直接拿去给 AI 或团队使用。 评分时看这些点：是否完整覆盖 6 个要求；是否紧扣 API 调用与安全 和 公开 Web 产品集成 OpenAI API Key；交付物是否具体、可执行、可复制；是否有明确质量检查；是否认真处理核心风险「前端泄露 Key 或账单暴增」；是否避免空话和泛泛而谈。

讲的时候提醒学员：不要只背关键词，要把关键词放回真实场景里。题目一换皮，场景理解才是拿分关键。

### Slide 6 — 小白最容易混淆的点

- 如果题目问“概念”，先给一句人话定义，再补英文术语。
- 如果题目问“场景”，先判断任务类型，再选工具或 Prompt 策略。
- 如果题目问“Debug”，先找根因，不要直接堆更多功能。
- 如果题目问“Boss 综合”，答案要有结构：目标、步骤、格式、边界、检查。

### Slide 7 — 本课实战模板

今天的开放题是：直接写 Prompt：围绕「公开 Web 产品集成 OpenAI API Key」，写一个可以直接复制给大模型使用的 API 调用与安全 Prompt。要求包含：角色/目标、输入信息、执行步骤、输出格式、边界限制。

可以按这个模板来答：

1. 先写清楚角色或目标。
2. 再写输入是什么、处理步骤是什么。
3. 明确输出格式，不要让 AI 自由发挥。
4. 加上边界条件：不知道怎么办、失败怎么办、哪些不能做。
5. 最后检查评分标准：结构完整、细节具体、能直接拿去用。
